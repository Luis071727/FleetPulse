# FleetPulse AI — Product Constitution
# This file defines the non-negotiable constraints, architectural decisions,
# and product philosophy for all AI-assisted build sessions.
# Do not override these decisions unless explicitly instructed.

## Product Identity

You are building FleetPulse AI — a vertical SaaS platform for small US trucking
carriers (owner-operators and fleets of 1–20 trucks). There are 3.5M+ of these
in the US. They are underserved by existing enterprise TMS tools.

The product has two distinct front-end applications sharing one backend:
  1. Dispatcher Command Center — full fleet management for the dispatcher
  2. Carrier Self-Service Portal — read-only personal dashboard per carrier

The dispatcher is the primary user and primary distribution channel.
The carrier is the secondary user and the upgrade revenue opportunity.

## Tech Stack — Do Not Substitute Without Explicit Approval

  Frontend:    Next.js 14 + Tailwind CSS → hosted on Vercel (free tier)
  Backend:     Python 3.12 + FastAPI     → hosted on Railway (free tier)
  Database:    Supabase (PostgreSQL)     → includes Auth + RLS + Realtime
  AI Engine:   Anthropic Claude Sonnet  → model: claude-sonnet-4-20250514
  ELD Data:    Terminal API (withterminal.com) — unified ELD across 50+ providers
  MVR Monitor: Embark Safety or Verified Credentials — continuous driver monitoring
  PSP Reports: FMCSA PSP Program — $10/pull on demand
  Public Data: FMCSA SAFER API — carrier + broker DOT/MC lookup (free, cached 24h)
  SMS:         Twilio
  Email:       SendGrid
  Payments:    Stripe (Checkout + webhooks)

Target infrastructure cost at MVP: under $30/month.

## Database Conventions

  - All tables use uuid PRIMARY KEY DEFAULT gen_random_uuid()
  - All timestamps are timestamptz DEFAULT now() in UTC
  - Row-Level Security (RLS) is ENABLED on every table — no exceptions
  - Dispatchers see all rows WHERE organization_id = their org
  - Carriers see ONLY their own rows WHERE carrier_id = their id
  - Soft deletes only — never hard-delete carrier or load records
  - Column naming: snake_case throughout

## API Conventions

  - All endpoints prefixed: /api/v1/
  - Auth: Supabase JWT via Authorization: Bearer 
  - Response envelope: { data, error, meta }
  - FMCSA responses cached in Supabase for 24h — never call FMCSA twice for same DOT
  - All AI calls are server-side only (FastAPI) — never call Claude from the frontend
  - Claude prompt caching enabled on all large carrier context blocks

## User Roles & Access

  dispatcher_admin  → sees ALL carriers in org, full edit, all AI features
  carrier_free      → read-only, own data only, no AI features (dispatcher pays)
  carrier_pro       → own data, load profitability, broker scores, invoice drafts ($29/mo)
  carrier_fleet     → own fleet, team access, full insurance module ($59/mo)

  Carrier portal access is invite-only. Dispatcher sends invite via SendGrid magic link.
  Carrier sets password on first login via Supabase email invite flow.

## Core Business Logic

  Load net profit = load_rate - fuel_cost - driver_pay - tolls - other_costs
  Load RPM        = load_rate / miles
  Load net RPM    = net_profit / miles

  AI load recommendation thresholds:
    GO         →  net_rpm >= 1.50 AND broker trust_score >= 70
    NEGOTIATE  →  net_rpm >= 1.00 OR broker trust_score 50-69
    PASS       →  net_rpm < 1.00 OR trust_score < 50 OR fraud_flags > 0

  Broker trust score (0-100) weighted factors:
    FMCSA authority status   30%
    Avg payment days         25%
    Late payment rate        20%
    Fraud/complaint flags    15%
    Operating history years  10%

  Insurance Readiness Score (IRS, 0-100) weighted sub-scores:
    Safety Record       25%  (CSA BASICs, crash history)
    Driver Quality      20%  (MVR scores, CDL, PSP)
    Compliance          20%  (ELD, HOS, drug testing)
    Fleet Risk Profile  15%  (truck age, radius verification)
    Safety Technology   10%  (dashcams, telematics)
    Market Readiness    10%  (FMCSA rating, no coverage lapse)

  Invoice follow-up tone:
    7-14 days   → polite reminder
    15-21 days  → firm follow-up
    22-29 days  → assertive with deadline
    30+ days    → final notice with escalation options

## Build Phase Sequence — Do Not Skip Phases

  Phase 1 (Weeks 1-7):   Dispatcher Command Center
    Week 1-2: Carrier roster + FMCSA integration
    Week 3-4: Load logging + invoice tracker
    Week 5-6: AI layer (broker scores, load analysis, invoice drafts)
    Week 7:   Carrier portal invites + portal MVP (read-only)

  Phase 2 (Weeks 8-12):  Insurance Intelligence
    IRS scoring engine, driver profiles, MVR integration,
    AI playbook generator, DataQs challenge letter drafting

  Phase 3 (Weeks 13-16): HOS Compliance + ELD
    Terminal API OAuth, HOS clock monitoring, Twilio SMS alerts,
    AI HOS contextualization ("driver has 1h52m, load delivers in 3h20m")

  Phase 4 (Weeks 17-20): Billing + Growth
    Stripe subscriptions, carrier upgrade flow, analytics/reports

  Current phase: Phase 1, Week 1.
  Do not build Phase 2+ features until Phase 1 gate is passed.
  Gate: dispatcher can add a real carrier and view their loads.

## AI Prompt Rules

  Model: claude-sonnet-4-20250514 (always Sonnet, never Haiku or Opus)
  Prompt caching: enabled on all system prompts with large carrier context
  All AI responses must be structured JSON — no markdown fences, no preamble
  Never call Claude from the frontend — only via FastAPI endpoints

  Four AI endpoint categories:
    /ai/load/analyze       → Go/Negotiate/Pass recommendation + net profit
    /ai/broker/score       → Trust score 0-100 + risk summary
    /ai/invoice/followup   → Follow-up message draft with tone + channel
    /ai/insurance/playbook → Prioritized improvement plan + savings estimate

## External Integration Constraints

  FMCSA SAFER:
    Rate limit: 100 req/min — implement 24h cache, never skip cache
    On 404: return { found: false } — do not throw error
    Always cache response in supabase table: fmcsa_cache

  Terminal API (ELD):
    Phase 3 only — do not implement in Phase 1 or 2
    OAuth 2.0 per carrier — dispatcher initiates, carrier authorizes
    Webhook required for HOS threshold events

  Embark Safety / MVR:
    Phase 2 only
    Driver must sign consent form before any MVR pull (FCRA requirement)
    Severity levels: 1=DUI/reckless (-20 IRS pts), 2=major speeding (-10), 3=minor (-5)

  Stripe:
    Webhooks required: checkout.session.completed, customer.subscription.updated,
    customer.subscription.deleted, invoice.payment_failed
    No free trials in Phase 1
    Carrier upgrade via Stripe Checkout (not custom form)

## Notification Rules

  CSA breach or MVR violation  → in-app + email + SMS to dispatcher
  Invoice 30+ days overdue     → in-app + email to dispatcher
  Renewal 60 days out          → in-app + email to dispatcher + carrier
  Renewal 30 days out          → in-app + email + SMS to dispatcher + carrier
  HOS 2hr remaining            → SMS to dispatcher only
  HOS 30min remaining          → SMS critical to dispatcher only
  Portal invite                → SendGrid magic link email to carrier

## Things That Are Decided — Do Not Re-Litigate

  - Mobile web (not native app) for carrier portal in Phase 1-3
  - Single organization per carrier in Phase 1 (no multi-dispatcher tenancy)
  - No free trial — sell on value
  - Dispatcher is the GTM entry point, not direct-to-carrier cold outreach
  - Soft deletes only on all user-facing records
  - All AI calls server-side only — never expose Claude API key to frontend
  - Prompt caching on all large context AI calls to hit 60%+ cache hit rate
